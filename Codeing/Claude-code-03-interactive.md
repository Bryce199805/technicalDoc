## 1. 交互模式基础

### 1.1 进入交互模式

在终端中直接输入 `claude` 即可进入交互式 REPL（Read-Eval-Print Loop）环境：

```bash
# 最基本的方式 — 在当前目录启动交互模式
claude

# 带初始提示进入
claude "帮我检查这个项目的结构"

# 在指定目录启动
cd /path/to/project && claude
```

启动后，你会看到类似如下的界面：

```
╭─────────────────────────────────────────╮
│ ✻ Claude Code                           │
│                                         │
│   /help for help                        │
╰─────────────────────────────────────────╯

 >
```

### 1.2 提示符界面说明

交互模式的提示符 `>` 是你与 Claude 对话的入口。界面包含以下关键元素：

- **提示符 `>`**：等待你输入指令或自然语言描述
- **顶部横幅**：显示 Claude Code 版本信息和基本提示
- **状态指示**：当 Claude 正在思考或执行操作时，会显示加载动画
- **工具调用面板**：Claude 使用工具时会显示具体的操作内容，你可以选择批准或拒绝

### 1.3 输入多行文本

默认情况下，按 `Enter` 键会直接发送消息。如果你需要输入多行文本：

```
方法一：Shift + Enter  （换行，不发送）
方法二：Option + Enter  （macOS，换行不发送）
方法三：直接粘贴多行文本 （粘贴的换行符会被保留）
```

**示例 — 输入多行代码片段让 Claude 分析：**

```
> 请分析以下代码的问题：          ← 输入第一行
  function getData() {             ← Shift+Enter 换行
    const result = fetch('/api')   ← Shift+Enter 换行
    return result                  ← Shift+Enter 换行
  }                                ← Enter 发送
```

> **提示**：如果你需要粘贴大段代码，直接从编辑器复制粘贴即可，Claude 会自动识别多行内容。

## 2. 斜杠命令完整列表

斜杠命令（Slash Commands）是交互模式中以 `/` 开头的特殊指令，用于控制 Claude Code 的行为。在提示符中输入 `/` 即可看到可用命令的自动补全列表。

### 2.1 `/help` — 显示帮助信息

```
> /help
```

显示所有可用的斜杠命令及其简要说明。这是你遇到困惑时的第一个求助对象。

输出内容包括：
- 所有可用斜杠命令的列表
- 每个命令的简要功能描述
- 基本使用提示

### 2.2 `/compact [instructions]` — 压缩上下文窗口

```
> /compact
> /compact 保留所有与数据库相关的讨论内容
> /compact focus on the API design decisions
```

当对话变长、上下文窗口接近上限时，`/compact` 是你的救星。它会智能压缩之前的对话内容，保留关键信息，释放上下文空间。

**核心特性：**
- 不带参数时，Claude 会自动判断哪些内容重要并保留
- 带参数时，你可以指导 Claude 在压缩过程中优先保留哪些内容
- 压缩后的摘要会替代原始对话，但关键决策和代码变更信息通常会被保留
- 压缩是不可逆的 — 原始对话细节会丢失

**使用场景：**
```
# 长时间调试后，压缩但保留错误信息
> /compact 保留所有错误堆栈和修复方案

# 架构讨论后，只保留最终决策
> /compact 只保留最终的架构决策和选型理由
```

### 2.3 `/clear` — 清空整个会话上下文

```
> /clear
```

与 `/compact` 不同，`/clear` 会**完全清除**所有对话历史，让你从零开始。这相当于重新启动一个全新的会话，但不需要退出再重新进入。

**注意事项：**
- 清除后无法恢复之前的对话内容
- CLAUDE.md 中的项目记忆不受影响（它会在新对话开始时重新加载）
- 适合在完成一个任务后、开始全新任务时使用

### 2.4 `/context` — 查看当前上下文使用情况

```
> /context
```

显示当前会话的上下文窗口使用情况，包括：

- 已使用的 token 数量
- 上下文窗口的总容量
- 使用百分比
- 剩余可用空间

**输出示例：**
```
Context usage:
  Used: 45,230 tokens (22.6%)
  Remaining: 154,770 tokens
  Total: 200,000 tokens
```

> **最佳实践**：当使用率超过 70% 时，建议使用 `/compact` 压缩上下文，避免因上下文溢出导致早期重要信息丢失。

### 2.5 `/model` — 切换 AI 模型

```
> /model
```

交互式选择要使用的 AI 模型。可选模型通常包括：

| 模型 | 特点 | 适用场景 |
|------|------|----------|
| **Claude Sonnet** | 速度快，性价比高 | 日常编码、快速问答、简单重构 |
| **Claude Opus** | 最强推理能力 | 复杂架构设计、疑难 bug 调试、多文件重构 |
| **Claude Haiku** | 最快响应速度 | 简单查询、格式转换、快速解释 |

```
# 切换模型
> /model
  ❯ claude-sonnet-4-20250514
    claude-opus-4-20250514
    claude-haiku-3-20250307
```

### 2.6 `/cost` — 显示当前会话费用统计

```
> /cost
```

实时显示本次会话的 API 费用消耗，帮助你追踪成本。输出信息包括：

- 本次会话总费用
- 输入 token 费用
- 输出 token 费用
- 各次交互的费用明细

### 2.7 `/plan` — 进入规划模式（Plan Mode）

```
> /plan
```

切换进入 Plan Mode（规划模式）。在此模式下，Claude 只进行分析和规划，**不会实际执行**任何文件修改操作。详见 [第 7 节 Plan Mode 详解](#7-plan-mode-详解)。

### 2.8 `/commit` — 自动生成 commit 并提交

```
> /commit
```

Claude 会自动：
1. 检查当前 `git status`，识别所有变更
2. 分析变更内容，理解修改的目的
3. 生成符合规范的 commit message
4. 执行 `git add` 和 `git commit`

**生成的 commit message 格式通常为：**
```
feat: 添加用户认证中间件

- 实现JWT token验证
- 添加角色权限检查
- 创建认证错误处理

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

> **提示**：你可以在提交前审查 Claude 生成的 commit message，选择接受或要求修改。

### 2.9 `/review-pr [number]` — 审查 Pull Request

```
> /review-pr          # 审查当前分支的PR
> /review-pr 42       # 审查指定编号的PR
```

Claude 会拉取 PR 的所有变更，进行全面审查并提供：
- 代码质量评估
- 潜在 bug 和安全问题
- 性能影响分析
- 改进建议

### 2.10 `/memory` — 编辑 CLAUDE.md 记忆文件

```
> /memory
```

打开并编辑项目的 `CLAUDE.md` 文件。这是 Claude Code 的"长期记忆"，存储项目特定的指令、规范和上下文信息。关于记忆系统的详细说明，请参阅下一章。

### 2.11 `/permissions` — 查看和管理权限设置

```
> /permissions
```

显示和管理 Claude Code 的工具权限。你可以查看：
- 当前已授权的工具列表
- 已拒绝的工具和命令
- 各权限的作用范围（会话级 / 项目级 / 全局级）

**权限级别：**
```
Allow always  → 始终允许（记住这个选择）
Allow once    → 仅本次允许
Deny always   → 始终拒绝
Deny once     → 仅本次拒绝
```

### 2.12 `/doctor` — 诊断配置和连接问题

```
> /doctor
```

当 Claude Code 工作异常时，运行 `/doctor` 进行自动诊断。检查项目包括：

- API 密钥配置是否正确
- 网络连接是否正常
- 模型访问权限是否有效
- 本地配置文件是否存在错误
- Git 环境是否正常
- 依赖工具是否安装

### 2.13 `/exit` — 退出 Claude Code

```
> /exit
```

优雅地退出交互模式。等同于按 `Ctrl+C` 两次或 `Ctrl+D`。

### 2.14 `/config` — 查看和修改配置

```
> /config
```

查看当前 Claude Code 的配置项，并可进行修改。常见配置包括：

- 默认模型选择
- 主题和外观设置
- 自动压缩行为
- 权限默认策略

### 2.15 `/status` — 查看状态信息

```
> /status
```

显示 Claude Code 的运行状态概览，包括：
- 当前使用的模型
- 会话 ID
- 上下文使用情况
- 认证状态
- 当前工作目录

### 2.16 `/bug` — 报告 Bug

```
> /bug
```

当你遇到 Claude Code 自身的问题时，使用此命令可以快速报告 bug。它会引导你描述问题，并自动收集相关的环境信息和日志。

### 2.17 `/init` — 初始化项目 CLAUDE.md

```
> /init
```

在当前项目中自动创建 `CLAUDE.md` 文件。Claude 会分析项目结构，自动生成包含以下内容的记忆文件：

- 项目技术栈识别
- 构建和测试命令
- 代码风格约定
- 目录结构说明

### 2.18 自定义 Skill 命令

除了内置斜杠命令外，Claude Code 还支持通过 Skill 系统注册自定义命令。这些命令以 `/` 开头，与内置命令使用方式一致。

```
# 例如，如果配置了自定义的 deploy skill
> /deploy staging

# 或者自定义的代码审查 skill
> /lint src/
```

自定义 Skill 的配置方式将在后续章节中详细介绍。

## 3. 键盘快捷键

掌握键盘快捷键可以大幅提升你在 Claude Code 交互模式中的效率。

### 3.1 基础快捷键一览

| 快捷键 | 功能 | 说明 |
|--------|------|------|
| `Enter` | 发送消息 | 提交当前输入的内容给 Claude |
| `Shift+Enter` | 换行 | 在输入中插入新行，不发送 |
| `Option+Enter` | 换行（macOS） | macOS 上的替代换行方式 |
| `Ctrl+C` | 取消/退出 | 取消当前操作，连按两次退出 |
| `Escape` | 中断生成 | 立即停止 Claude 的当前输出 |
| `Tab` | 自动补全 | 补全文件路径和斜杠命令 |
| `↑` / `↓` | 浏览历史 | 在历史输入命令之间切换 |

### 3.2 详细说明

#### `Enter` — 发送消息

按下 `Enter` 会立即将输入框中的内容发送给 Claude。确保你的消息已经完整再按发送。

#### `Shift+Enter` / `Option+Enter` — 换行

当你需要输入多行内容时使用。特别适合：
- 粘贴代码片段前添加说明文字
- 输入结构化的指令
- 编写多步骤的任务描述

#### `Ctrl+C` — 取消当前操作

```
场景一：Claude 正在执行操作 → Ctrl+C 取消操作
场景二：输入提示符处 → Ctrl+C 清空当前输入
场景三：连续按两次 Ctrl+C → 退出 Claude Code
```

#### `Escape` — 中断当前生成

当 Claude 正在生成回复时，按 `Escape` 可以立即中断。已生成的部分内容会保留在对话中。这在以下情况特别有用：
- Claude 的回答方向偏离了你的期望
- 你想修改或补充你的问题
- 响应太长，你已经得到了需要的信息

#### `Tab` — 自动补全文件路径

在输入中按 `Tab` 键会触发智能补全：

```
> 请查看 src/com[Tab]
  → src/components/
  → src/common/
  → src/config/
```

支持补全的内容：
- 文件和目录路径
- 斜杠命令（输入 `/` 后按 Tab）

#### `↑` / `↓` 箭头 — 浏览历史命令

与终端的历史命令功能类似，按上下箭头可以快速调出之前输入过的消息，方便你修改后重新发送。

## 4. 会话管理

Claude Code 的会话管理系统允许你保存、恢复和管理多个对话上下文。

### 4.1 继续最近的会话

```bash
# 使用完整参数
claude --continue

# 使用简写
claude -c
```

此命令会恢复你最近一次退出的会话，包括所有对话历史和上下文。适合以下场景：
- 意外关闭了终端
- 暂时离开后想继续之前的工作
- 昨天的工作今天想接着做

### 4.2 选择并恢复历史会话

```bash
# 使用完整参数
claude --resume

# 使用简写
claude -r
```

显示一个交互式列表，列出所有保存的历史会话，你可以选择要恢复的那一个：

```
? Select a session to resume:
  ❯ [2024-03-06 14:23] 重构用户认证模块
    [2024-03-06 10:15] 修复支付接口bug
    [2024-03-05 16:42] 设计新的API架构
    [2024-03-05 09:30] 编写单元测试
```

### 4.3 指定会话 ID

```bash
claude --session-id abc123def456
```

直接通过会话 ID 恢复特定会话。会话 ID 可以在 `/status` 命令的输出中找到。适合：
- 在脚本中自动化恢复特定会话
- 精确定位到某个特定的工作会话
- 与团队成员共享会话引用

### 4.4 分叉会话

```bash
claude --resume --fork-session
```

`--fork-session` 参数会在恢复会话时创建一个独立的分支副本。原始会话保持不变，你在分叉会话中的所有操作都不会影响原始会话。

**适用场景：**
- 想基于之前的讨论探索不同的方案
- 在不破坏原始上下文的情况下尝试新思路
- 从某个决策点出发，分别探索两个方向

```bash
# 实际使用示例
# 恢复之前的架构讨论，创建分叉来探索方案B
claude --resume --fork-session

> 之前我们讨论了微服务方案，现在让我们在这个分叉中探索单体架构方案
```

### 4.5 会话存储位置和管理

Claude Code 的会话数据存储在本地：

```
~/.claude/
├── sessions/          # 会话数据目录
│   ├── <session-id>/  # 每个会话一个目录
│   │   ├── history    # 对话历史
│   │   └── metadata   # 会话元数据
├── settings.json      # 全局配置
└── credentials        # 认证信息
```

**会话管理建议：**
- 定期清理不再需要的旧会话，释放磁盘空间
- 重要的会话可以通过 `--session-id` 记录下来以便后续恢复
- 会话不会自动过期，需要手动管理

## 5. 模型切换与快速模式

### 5.1 使用 `/model` 命令切换

在交互模式中随时切换模型：

```
> /model

? Select a model:
  ❯ claude-sonnet-4-20250514     (推荐日常使用)
    claude-opus-4-20250514       (复杂任务)
    claude-haiku-3-20250307      (快速响应)
```

切换后，后续所有交互都会使用新选择的模型，直到你再次切换。上下文历史不受影响。

### 5.2 启动时指定模型

```bash
# 使用 --model flag
claude --model claude-opus-4-20250514

# 简写形式
claude --model sonnet
claude --model opus
claude --model haiku
```

### 5.3 快速模式 vs 标准模式

Claude Code 中存在两种操作模式，适用于不同场景：

**标准模式（默认）：**
- 使用完整的推理能力
- 每次工具调用前需要你的确认（除非已配置自动批准）
- 适合需要精确控制的操作

**快速模式：**
- 优先使用速度更快的模型
- 减少不必要的确认步骤
- 适合你对操作结果有信心时使用

### 5.4 模型选择策略

根据任务类型选择最合适的模型：

```
日常编码任务 ──────────────→ Sonnet（最佳平衡）
  ├── 写函数、修bug、添加功能
  ├── 代码解释和问答
  └── 简单重构

复杂推理任务 ──────────────→ Opus（最强能力）
  ├── 多文件架构重构
  ├── 复杂算法设计
  ├── 跨模块依赖分析
  └── 疑难bug深度调试

快速查询任务 ──────────────→ Haiku（最快速度）
  ├── 语法查询
  ├── 简单格式转换
  └── 快速代码解释
```

> **费用提示**：Opus 的费用显著高于 Sonnet，Haiku 最为经济。日常使用推荐 Sonnet，仅在真正需要深度推理时切换到 Opus。

## 6. 上下文管理

上下文管理是高效使用 Claude Code 的关键技能。理解并善用上下文窗口，可以避免信息丢失、降低费用、提升响应质量。

### 6.1 上下文窗口概念

Claude 的上下文窗口是一个**有限的滑动窗口**，包含了当前对话的所有内容：

```
┌─────────────────────────────────────────────┐
│              上下文窗口（200K tokens）         │
├─────────────────────────────────────────────┤
│ 系统提示 + CLAUDE.md          ～2K tokens    │
│ 你的第1条消息                  ～200 tokens   │
│ Claude的第1条回复              ～800 tokens   │
│ 工具调用及结果                 ～3K tokens    │
│ 你的第2条消息                  ～150 tokens   │
│ Claude的第2条回复              ～1.2K tokens  │
│ ...                                         │
│ [已使用: 45K / 200K]                         │
│ [剩余: 155K tokens]                          │
└─────────────────────────────────────────────┘
```

当上下文窗口被占满时，最早的内容会被自动裁剪，可能导致重要信息丢失。

### 6.2 `/compact` 智能压缩

`/compact` 命令使用 AI 对当前对话进行智能摘要：

```
压缩前：45,000 tokens（22.5%）
  │
  │  /compact 保留架构决策和API设计
  ▼
压缩后：8,000 tokens（4%）
  ✓ 保留了关键决策
  ✓ 保留了代码变更摘要
  ✗ 丢失了讨论细节
  ✗ 丢失了中间尝试过程
```

### 6.3 `/clear` 完全清除

```
> /clear
```

将上下文完全重置为初始状态，仅保留系统提示和 CLAUDE.md 内容。

### 6.4 自动压缩机制

当上下文使用量接近上限时，Claude Code 会**自动触发压缩**：

- 触发阈值：通常在上下文使用率达到 ~95% 时
- 自动压缩会保留最近的对话和关键信息
- 你会收到通知，告知发生了自动压缩

### 6.5 最佳实践：何时压缩，何时清除

```
场景                        → 推荐操作
─────────────────────────────────────────
上下文超过70%，同一任务     → /compact
完成一个任务，开始新任务    → /clear
调试陷入死循环              → /clear + 重新描述问题
长时间架构讨论后要写代码    → /compact 保留决策要点
多次尝试失败                → /clear + 换个思路重新开始
```

**黄金法则：**
- 上下文不是越多越好 — 过多的无关信息会降低 Claude 的响应质量
- 定期使用 `/context` 检查使用量
- 与其让上下文自动溢出裁剪，不如主动 `/compact` 控制保留内容
- 新任务永远从 `/clear` 开始

## 7. Plan Mode 详解

### 7.1 什么是 Plan Mode

Plan Mode（规划模式）是 Claude Code 的一种特殊工作模式。在这个模式下，Claude 会**只进行分析、规划和讨论**，而不会实际执行任何文件操作（不写文件、不运行命令、不修改代码）。

这类似于在动手之前先画蓝图 — 确保方向正确后再执行。

### 7.2 进入和退出 Plan Mode

```
# 进入 Plan Mode
> /plan

# 退出 Plan Mode（回到正常执行模式）
> /plan
```

Plan Mode 是一个切换开关，再次输入 `/plan` 即可退出。进入 Plan Mode 后，提示符会有视觉提示，表明当前处于规划模式。

### 7.3 Plan Mode 工作流

Plan Mode 的典型工作流程遵循以下步骤：

```
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│  探索    │ ──→ │  规划    │ ──→ │  审批    │ ──→ │  执行    │
│ Explore  │     │  Plan    │     │ Review   │     │ Execute  │
└─────────┘     └─────────┘     └─────────┘     └─────────┘
    ↑                                                  │
    └──────────── 迭代反馈 ────────────────────────────┘
```

**第一步：探索（Plan Mode）**
```
> /plan
> 我想给这个项目添加用户认证功能，请先分析一下现有代码结构
```
Claude 会读取代码、分析项目结构，但不做任何修改。

**第二步：规划（Plan Mode）**
```
> 基于你的分析，请制定一个详细的实施计划
```
Claude 会输出分步骤的实施方案，包括要修改的文件、新增的文件、具体的代码变更等。

**第三步：审批（你来决定）**
```
> 计划看起来不错，但第3步我想用JWT而不是Session，请调整
```
你可以逐步审查计划，提出修改意见，直到满意为止。

**第四步：执行（退出 Plan Mode）**
```
> /plan          ← 退出 Plan Mode
> 请按照刚才的计划开始执行
```
Claude 开始实际执行操作，按照已审批的计划修改代码。

### 7.4 适用场景

Plan Mode 在以下场景中特别有价值：

**复杂功能开发：**
```
> /plan
> 我需要在这个电商系统中添加优惠券功能。
  需要支持：满减券、折扣券、限时券。
  请分析需要修改哪些模块，给出完整方案。
```

**架构变更：**
```
> /plan
> 我想把这个单体应用的用户模块拆分成微服务。
  请分析当前的耦合点，制定迁移计划。
```

**多文件修改：**
```
> /plan
> 我需要把项目的状态管理从Redux迁移到Zustand。
  请列出所有需要修改的文件和具体变更。
```

### 7.5 与直接执行的区别

| 对比项 | 直接执行模式 | Plan Mode |
|--------|-------------|-----------|
| 文件修改 | 实际执行修改 | 仅描述计划修改 |
| 命令执行 | 运行实际命令 | 仅说明要运行什么 |
| 风险等级 | 较高（直接改代码） | 零风险（只是讨论） |
| 适合场景 | 简单、明确的任务 | 复杂、需要规划的任务 |
| 反馈成本 | 回滚代码变更 | 修改计划文字即可 |

> **建议**：对于影响超过 3 个文件的变更，或者你不确定最佳方案时，优先使用 Plan Mode。

## 8. Checkpoint 回溯功能

### 8.1 Checkpoint 是什么

Checkpoint（检查点）是 Claude Code 在执行操作过程中自动创建的**代码状态快照**。它基于 Git 的功能实现，让你可以安全地回退到之前的任意状态。

可以把 Checkpoint 想象成游戏中的"存档点" — 如果 Claude 的操作结果不理想，你可以随时"读档"回到之前的状态。

### 8.2 自动创建 Checkpoint

Claude Code 会在以下时机自动创建 Checkpoint：

```
时机                              说明
─────────────────────────────────────────────
每次文件修改前                    保存修改前的代码状态
批量操作的开始                    多文件修改前的整体快照
关键操作前                        如删除文件、重大重构前
```

Checkpoint 的创建是透明的，不需要你手动操作。你可以在 Claude 的输出中看到 Checkpoint 创建的提示。

### 8.3 回溯到之前的状态

当 Claude 的修改不符合预期时，你可以通过对话中显示的 Checkpoint 标记来回溯：

```
> 把代码回滚到上一个 checkpoint

# 或者更具体地
> 回到你修改 auth.ts 之前的状态
```

Claude 会使用 Git 操作将代码恢复到指定的 Checkpoint 状态。

**回溯后的注意事项：**
- 代码文件会恢复到 Checkpoint 时的状态
- 对话历史仍然保留（你可以看到之前的操作记录）
- 你可以在回溯后给出新的指令，让 Claude 用不同的方式重做

### 8.4 使用场景

```
场景一：修改效果不佳
  Claude修改了代码 → 测试失败 → 回溯到Checkpoint → 用新思路重做

场景二：探索性开发
  尝试方案A → 回溯 → 尝试方案B → 比较效果 → 选择更好的方案

场景三：误操作恢复
  Claude误删了重要代码 → 立即回溯 → 代码完整恢复

场景四：渐进式开发
  第一步修改 → Checkpoint ✓
  第二步修改 → Checkpoint ✓
  第三步修改 → 出错 → 回溯到第二步 → 重做第三步
```

> **重要**：Checkpoint 依赖于 Git，因此只在 Git 仓库中有效。确保你的项目已经初始化了 Git。

## 9. Todo/Task 系统

### 9.1 什么是 Task 系统

Claude Code 内置了一个任务管理系统，用于跟踪和管理复杂工作流中的多个子任务。当一个大任务需要拆分成多个步骤时，Task 系统可以帮助 Claude 有条理地逐步完成。

### 9.2 核心操作

Task 系统提供四个核心操作：

#### TaskCreate — 创建任务

Claude 可以将复杂任务拆解为结构化的子任务列表：

```
任务：重构用户认证系统
├── [ ] 1. 分析现有认证代码
├── [ ] 2. 设计新的认证架构
├── [ ] 3. 实现JWT token服务
├── [ ] 4. 更新用户模型
├── [ ] 5. 修改登录/注册接口
├── [ ] 6. 添加中间件
├── [ ] 7. 编写单元测试
└── [ ] 8. 更新API文档
```

#### TaskUpdate — 更新任务状态

随着工作推进，Claude 会自动更新任务状态：

```
任务：重构用户认证系统
├── [✓] 1. 分析现有认证代码
├── [✓] 2. 设计新的认证架构
├── [→] 3. 实现JWT token服务     ← 当前正在进行
├── [ ] 4. 更新用户模型
├── [ ] 5. 修改登录/注册接口
├── [ ] 6. 添加中间件
├── [ ] 7. 编写单元测试
└── [ ] 8. 更新API文档
```

#### TaskList — 查看所有任务

获取当前会话中所有任务的概览，包括各任务的状态和进度。

#### TaskGet — 获取任务详情

查看特定任务的详细信息，包括描述、状态、依赖关系和执行记录。

### 9.3 任务状态管理

每个任务可以处于以下状态：

| 状态 | 符号 | 含义 |
|------|------|------|
| Pending | `[ ]` | 等待开始 |
| In Progress | `[→]` | 正在执行中 |
| Completed | `[✓]` | 已完成 |
| Failed | `[✗]` | 执行失败 |
| Skipped | `[-]` | 已跳过 |

### 9.4 任务依赖关系

Task 系统支持任务之间的依赖关系，确保执行顺序正确：

```
任务依赖图：
  1. 分析代码 ─────┐
                    ├──→ 3. 实现JWT服务 ──→ 5. 修改接口 ──→ 7. 写测试
  2. 设计架构 ─────┘                    ↘
                                         4. 更新模型 ──→ 6. 添加中间件 ──→ 8. 更新文档
```

有依赖关系的任务会在其前置任务完成后才开始执行。

### 9.5 何时使用任务系统

**适合使用 Task 系统的场景：**
- 涉及多个步骤的复杂功能开发
- 需要特定执行顺序的操作序列
- 跨多个文件的系统性修改
- 需要进度追踪的长时间任务

**不需要 Task 系统的场景：**
- 简单的单文件修改
- 快速问答和代码解释
- 一步就能完成的操作

**使用示例：**
```
> 帮我把这个Express项目迁移到Fastify框架。
  请先创建任务计划，然后逐步执行。

Claude 会：
1. 创建完整的迁移任务列表
2. 分析依赖关系
3. 按顺序逐个执行
4. 实时更新进度
5. 遇到问题时暂停并询问你的意见
```

> **提示**：你可以在任何时候要求 Claude "显示当前任务进度"来查看整体执行状况。

## 小结

交互模式是 Claude Code 最核心的使用方式。掌握本章介绍的内容后，你应该能够：

- **高效对话**：使用多行输入、历史命令、快捷键与 Claude 流畅交互
- **精准控制**：通过斜杠命令控制 Claude 的行为和模式
- **管理会话**：保存、恢复、分叉会话，在不同任务间无缝切换
- **优化上下文**：合理使用压缩和清除，保持上下文窗口的健康状态
- **规划先行**：使用 Plan Mode 在执行前制定周密计划
- **安全回溯**：利用 Checkpoint 功能安全地尝试和回退
- **任务管理**：使用 Task 系统有条理地完成复杂工作流
