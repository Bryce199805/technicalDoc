# 01 虚拟化原理

---

## 1.1 什么是虚拟化

虚拟化是在一台物理机（宿主机）上，通过软件层模拟出多台独立的逻辑计算机（虚拟机），每台虚拟机拥有自己的 CPU、内存、磁盘、网卡，运行自己的操作系统。

```
┌─────────────────────────────────────────────┐
│           虚拟机1 (Kali)  虚拟机2 (Win11)    │
│           Guest OS        Guest OS           │
├─────────────────────────────────────────────┤
│              虚拟化管理层 (Hypervisor)         │
├─────────────────────────────────────────────┤
│           Linux 宿主机 (Ubuntu)               │
│           Host OS + KVM 内核模块              │
├─────────────────────────────────────────────┤
│           物理硬件 (CPU/内存/磁盘/网卡)         │
└─────────────────────────────────────────────┘
```

---

## 1.2 Hypervisor 类型

| 类型 | 代表 | 特点 |
|------|------|------|
| **Type 1（裸金属）** | VMware ESXi, Proxmox VE, Hyper-V | 直接跑在硬件上，没有宿主 OS |
| **Type 2（托管型）** | VirtualBox, VMware Workstation | 跑在宿主 OS 之上 |
| **KVM** | 特殊 | 编译进 Linux 内核，性能接近 Type 1，管理上像 Type 2 |

## 1.3 KVM：内核级虚拟化

### KVM 是什么

KVM（Kernel-based Virtual Machine）于 2006 年被合并进 Linux 内核（2.6.20），它把 Linux 内核本身变成了一个 Hypervisor。

核心思想：**让虚拟机直接跑在物理 CPU 上，内核只负责调度和资源分配。**

### KVM 做什么和不做什么

```
KVM 实际做的事：
  ✓ 把 CPU 的虚拟化指令暴露给虚拟机
  ✓ 管理 VM 的内存映射
  ✓ 处理 VM 的中断和异常
  ✓ 提供内核接口 (/dev/kvm)

KVM 不做的事（由 QEMU 负责）：
  ✗ 模拟磁盘、网卡、显卡等设备
  ✗ 提供 VM 的创建/启停管理界面
  ✗ 提供快照、迁移等高级功能
```

### KVM 内核模块

KVM 由两个内核模块组成：

| 模块 | 说明 |
|------|------|
| `kvm.ko` | 核心模块，提供虚拟化基础设施 |
| `kvm_intel.ko` | Intel CPU 专用，使用 VT-x 技术 |
| `kvm_amd.ko` | AMD CPU 专用，使用 AMD-V 技术 |

查看是否加载：

```bash
lsmod | grep kvm
# 输出示例：
# kvm_intel    364696  0
# kvm         1056352  1 kvm_intel
```

手动加载：

```bash
sudo modprobe kvm
sudo modprobe kvm_intel   # Intel CPU
sudo modprobe kvm_amd     # AMD CPU
```

### VM Exit 机制

当虚拟机执行了不能直接在物理 CPU 上运行的指令时，CPU 会触发 VM Exit，把控制权交回 KVM：

```
Guest OS 运行中
    │
    ├─ 普通指令 → CPU 直接执行 ✅（接近原生速度）
    ├─ I/O 请求 → VM Exit → QEMU 模拟设备返回结果 → VM Entry 回到 Guest
    └─ 特权指令 → VM Exit → KVM 拦截处理 → VM Entry 回到 Guest
```

**VM Exit 越少，性能越好。** 这也是 virtio 半虚拟化驱动比全虚拟化快的原因——它减少了不必要的 VM Exit。

## 1.4 QEMU：设备模拟器

### QEMU 是什么

QEMU 是一个纯软件的机器模拟器，可以不依赖 KVM 独立运行（但极慢，所有指令都靠软件模拟）。

### QEMU + KVM 组合

当 QEMU + KVM 组合时：

```
虚拟机中的普通指令 → 直接在物理 CPU 上执行（KVM 促成，接近原生速度）
虚拟机中的 I/O 操作 → QEMU 模拟设备，转为宿主机的系统调用
虚拟机中的特权指令 → 触发 VM Exit，由 KVM 拦截处理
```

这就是 QEMU/KVM 组合能达到 95%+ 原生性能的原因。

### QEMU 进程模型

每个虚拟机对应一个 QEMU 进程：

```bash
# 查看运行中的 QEMU 进程
ps aux | grep qemu

# 输出示例：
# libvirt+  12345  2.5  10.0 ... qemu-system-x86_64 -name kali ...
```

- QEMU 进程以 `libvirt-qemu` 用户身份运行
- 因此 QEMU 无法直接读取 home 目录下的文件
- ISO 和磁盘镜像应放在 `/var/lib/libvirt/` 下

## 1.5 libvirt：统一管理层

### 为什么需要 libvirt

如果只用 QEMU/KVM，你需要写很长的命令行来创建虚拟机。libvirt 提供统一的 API 和管理界面。

### 架构

```
┌──────────┐  ┌──────────┐  ┌──────────┐
│  virsh   │  │ virt-    │  │ Python   │
│  命令行   │  │ install  │  │ libvirt  │
└────┬─────┘  └────┬─────┘  └────┬─────┘
     │              │              │
     └──────────────┼──────────────┘
                    │
            ┌───────▼────────┐
            │   libvirtd      │  ← 统一守护进程
            │   (API 层)      │
            └───────┬────────┘
                    │
     ┌──────────────┼──────────────┐
     ▼              ▼              ▼
  QEMU/KVM      LXC           Xen
  (全虚拟化)   (容器)       (另一种Hypervisor)
```

### libvirt 的功能

| 功能 | 说明 |
|------|------|
| 统一 API | 不管底层是 QEMU、LXC 还是 Xen，接口一致 |
| XML 配置 | 虚拟机配置用 XML 描述，可版本控制 |
| 网络管理 | NAT、桥接、隔离网络的自动管理 |
| 存储管理 | qcow2、raw、LVM 卷的管理 |
| 快照 | 创建、恢复、删除虚拟机快照 |
| 安全 | 自动生成 AppArmor profile、seccomp 沙箱 |

### 系统实例 vs 会话实例

| | 系统实例 (qemu:///system) | 会话实例 (qemu:///session) |
|---|---|---|
| 运行身份 | root (libvirtd 系统服务) | 当前用户 (用户级 libvirtd) |
| 网络功能 | 有 NAT/桥接/隔离网络，自动管理 | 没有默认网络，需手动配 |
| 存储位置 | `/var/lib/libvirt/` | `~/.local/share/libvirt/` |
| 开机自启 | VM 可随系统启动 | 不行，用户登录后才可用 |
| 适合场景 | 服务器、长期运行的 VM | 个人临时实验 |

**推荐使用系统实例**，网络开箱即用，支持开机自启。

## 1.6 virtio：半虚拟化驱动

### 全虚拟化 vs 半虚拟化

| | 全虚拟化 | 半虚拟化 (virtio) |
|---|---|---|
| 原理 | QEMU 完全模拟真实硬件 | Guest OS 知道自己运行在虚拟机中，配合宿主优化 |
| 性能 | 较差（大量 VM Exit） | 接近原生（减少 VM Exit） |
| 兼容性 | 无需 Guest 驱动 | 需要安装 Guest 驱动 |
| 示例 | IDE 磁盘、e1000 网卡 | virtio-blk 磁盘、virtio-net 网卡 |

### virtio 设备类型

| 设备 | 全虚拟化 | 半虚拟化 (virtio) |
|------|---------|-------------------|
| 磁盘 | `bus=ide` 或 `bus=sata` | `bus=virtio` |
| 网卡 | `model=e1000` 或 `model=rtl8139` | `model=virtio` |
| 串口 | 默认 | `target_type=serial` |
| 随机数 | 无 | `rng model=virtio` |
| 内存气球 | 无 | `model=virtio-balloon`（动态调整内存） |

**建议**：所有设备尽量使用 virtio，性能差异显著。

## 1.8 关键概念速查

| 概念 | 一句话解释 |
|------|-----------|
| **KVM** | 内核模块，让 VM 直接跑在物理 CPU 上 |
| **QEMU** | 用户态进程，模拟硬件设备（磁盘/网卡/显卡） |
| **libvirt** | 管理 API/守护进程，统一管理 QEMU/LXC/Xen |
| **virsh** | libvirt 的命令行客户端 |
| **virt-install** | 创建虚拟机的命令行工具 |
| **qcow2** | QEMU 的磁盘格式，支持精简分配和快照 |
| **virtio** | 半虚拟化驱动，Guest 配合宿主优化 I/O 性能 |
| **VM Exit** | VM 让出 CPU 控制权给宿主内核的过程 |
| **vCPU** | 虚拟 CPU，宿主机调度给 VM 使用的 CPU 时间片 |
| **Hypervisor** | 虚拟机监视器，管理和运行虚拟机的软件层 |
