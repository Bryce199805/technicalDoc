# Docker 基础概念

## 什么是 Docker

Docker 是一个开源的应用容器引擎，基于 Go 语言开发，遵从 Apache2.0 协议开源。Docker 可以让开发者将应用程序及其依赖打包到一个可移植的容器中，然后发布到任何流行的 Linux 或 Windows 机器上，也可以实现虚拟化。

### Docker 的核心价值

1. **一致性**：确保应用程序在任何环境中运行一致
2. **轻量级**：容器共享主机内核，启动速度快，资源占用少
3. **可移植性**：一次构建，到处运行
4. **隔离性**：应用程序之间相互隔离，互不影响
5. **版本控制**：镜像可以版本化管理，便于回滚

## Docker 架构

Docker 采用客户端-服务器（C/S）架构，使用远程 API 来管理和创建 Docker 容器。

```
┌─────────────────────────────────────────────────────────┐
│                      Docker Host                         │
│  ┌───────────────────────────────────────────────────┐  │
│  │                  Docker Daemon                     │  │
│  │              (dockerd - REST API)                  │  │
│  └───────────────────────────────────────────────────┘  │
│                          ▲                               │
│                          │ REST API                      │
│                          ▼                               │
│  ┌───────────────────────────────────────────────────┐  │
│  │                  Docker Client                     │  │
│  │                   (docker)                         │  │
│  └───────────────────────────────────────────────────┘  │
│                                                          │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐       │
│  │  Container  │ │  Container  │ │  Container  │       │
│  └─────────────┘ └─────────────┘ └─────────────┘       │
│                                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │                    Images                        │    │
│  └─────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
```

### 核心组件

| 组件 | 说明 |
|------|------|
| **Docker Client** | 用户与 Docker 交互的主要方式，通过命令行或 API 与 Docker Daemon 通信 |
| **Docker Daemon** | 后台服务进程，负责构建、运行和分发 Docker 容器 |
| **Docker Images** | 只读模板，包含创建 Docker 容器的指令 |
| **Docker Containers** | 镜像的运行实例，可以被启动、停止、删除 |
| **Docker Registry** | 存储 Docker 镜像的仓库，如 Docker Hub、私有仓库 |

## 核心概念详解

### 1. 镜像（Image）

镜像是一个只读的文件包，包含了运行应用所需的所有内容：代码、运行时、库、环境变量和配置文件。

#### 镜像的特点

```
┌─────────────────────────────┐
│         Application         │  ← 应用层
├─────────────────────────────┤
│         Dependencies        │  ← 依赖层
├─────────────────────────────┤
│         Runtime             │  ← 运行时层
├─────────────────────────────┤
│         Base OS             │  ← 基础系统层
└─────────────────────────────┘
```

- **分层存储**：镜像由多个只读层叠加而成，每层代表 Dockerfile 中的一条指令
- **UnionFS**：使用联合文件系统，多个层可以挂载到同一个挂载点
- **Copy-on-Write**：当容器修改文件时，会在容器层创建副本，不影响镜像层
- **镜像摘要**：每个镜像有唯一的 SHA256 标识

#### 镜像命名规范

```
[registry/][username/]repository[:tag][@digest]

示例：
nginx:latest                    # 官方镜像，latest 标签
nginx:1.21-alpine              # 官方镜像，指定标签
library/nginx:latest           # 完整写法，library 是默认命名空间
docker.io/library/nginx:latest # 完整仓库地址
myregistry.com/myapp:v1.0      # 私有仓库镜像
username/nginx:v1.0            # 用户命名空间下的镜像
```

#### 常用基础镜像选择

| 基础镜像 | 大小 | 适用场景 |
|---------|------|---------|
| `scratch` | 0 MB | 空镜像，适用于静态编译的 Go 程序 |
| `alpine` | ~5 MB | 最小化 Linux 发行版，适合生产环境 |
| `busybox` | ~1 MB | 嵌入式工具集，适合极简场景 |
| `debian:slim` | ~80 MB | Debian 精简版，兼容性好 |
| `ubuntu` | ~70 MB | 用户友好的 Linux 发行版 |
| `centos` | ~200 MB | 企业级 Linux 发行版 |

### 2. 容器（Container）

容器是镜像的运行实例，是一个独立运行的软件单元。

#### 容器的生命周期

```
         ┌──────────┐
         │ Created  │  ← docker create
         └────┬─────┘
              │ docker start
              ▼
         ┌──────────┐
    ┌───▶│ Running  │  ← docker run = create + start
    │    └────┬─────┘
    │         │ docker stop / docker kill
    │         ▼
    │    ┌──────────┐
    │    │ Stopped  │  ← docker rm 删除
    │    └──────────┘
    │         │ docker start
    └─────────┘
              │ docker pause
              ▼
         ┌──────────┐
         │ Paused   │  ← docker unpause 恢复
         └──────────┘
```

#### 容器的状态

| 状态 | 说明 | 命令 |
|------|------|------|
| `created` | 已创建但未启动 | `docker create` |
| `running` | 正在运行 | `docker start` / `docker run` |
| `paused` | 已暂停 | `docker pause` |
| `stopped` / `exited` | 已停止 | `docker stop` / `docker kill` |
| `restarting` | 正在重启 | 重启策略触发 |
| `removing` | 正在删除 | `docker rm` |
| `dead` | 已死亡 | 通常是由于磁盘空间不足 |

#### 容器与镜像的关系

```
┌─────────────────────────────────────────────────────────┐
│                      Container                          │
│  ┌─────────────────────────────────────────────────┐   │
│  │            Container Layer (R/W)                 │   │
│  │         可写层 - 存储容器修改                      │   │
│  └─────────────────────────────────────────────────┘   │
├─────────────────────────────────────────────────────────┤
│                        Image                            │
│  ┌─────────────────────────────────────────────────┐   │
│  │              Layer N (R/O)                       │   │
│  ├─────────────────────────────────────────────────┤   │
│  │              Layer 2 (R/O)                       │   │
│  ├─────────────────────────────────────────────────┤   │
│  │              Layer 1 (R/O)                       │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 3. 仓库（Registry）

仓库是集中存放镜像的地方，分为公有仓库和私有仓库。

#### 常用镜像仓库

| 仓库 | 类型 | 说明 |
|------|------|------|
| [Docker Hub](https://hub.docker.com) | 公有 | Docker 官方仓库，最丰富 |
| [Quay.io](https://quay.io) | 公有 | Red Hat 运营 |
| [GitHub Container Registry](https://ghcr.io) | 公有 | GitHub 托管的容器仓库 |
| [阿里云容器镜像服务](https://cr.console.aliyun.com) | 公有/私有 | 国内访问速度快 |
| [腾讯云容器镜像服务](https://cloud.tencent.com/product/tcr) | 公有/私有 | 国内访问速度快 |
| Harbor | 私有 | 企业级私有仓库 |
| Nexus | 私有 | 支持多种制品管理 |
| GitLab Container Registry | 私有 | GitLab 内置 |

#### 镜像仓库操作

```bash
# 登录仓库
docker login registry.example.com

# 登出仓库
docker logout registry.example.com

# 推送镜像
docker push registry.example.com/username/image:tag

# 拉取镜像
docker pull registry.example.com/username/image:tag
```

### 详细对比

| 特性 | Docker 容器 | 虚拟机 |
|------|------------|--------|
| **启动速度** | 秒级 | 分钟级 |
| **资源占用** | MB 级别 | GB 级别 |
| **性能** | 接近原生 | 有损耗 |
| **系统支持量** | 单机支持上千个容器 | 单机几十个虚拟机 |
| **隔离性** | 进程级别隔离 | 操作系统级别隔离 |
| **安全性** | 相对较弱（共享内核） | 较强（独立内核） |
| **操作系统** | 与宿主机共享内核 | 独立操作系统 |
| **迁移性** | 轻量，快速迁移 | 较大，迁移较慢 |

### 适用场景

| 场景 | 推荐 |
|------|------|
| 微服务架构 | Docker |
| 持续集成/持续部署 | Docker |
| 开发环境标准化 | Docker |
| 高密度部署 | Docker |
| 需要完整操作系统隔离 | 虚拟机 |
| 运行不同内核的系统 | 虚拟机 |
| 高安全隔离要求 | 虚拟机 |

## Docker 的底层技术

### 1. Namespaces（命名空间）

命名空间为容器提供隔离，Docker 使用以下命名空间：

| Namespace | 作用 | 隔离内容 |
|-----------|------|---------|
| `pid` | 进程隔离 | 进程 ID |
| `net` | 网络隔离 | 网络设备、协议栈、端口 |
| `ipc` | IPC 隔离 | 信号量、消息队列 |
| `mnt` | 文件系统隔离 | 挂载点 |
| `uts` | 主机名隔离 | 主机名和域名 |
| `user` | 用户隔离 | 用户和用户组 |
| `cgroup` | CGroup 隔离 | CGroup 根目录 |

### 2. CGroups（控制组）

CGroups 用于限制、记录和隔离进程组使用的物理资源：

```
┌─────────────────────────────────────────┐
│              CGroups                     │
├─────────────────────────────────────────┤
│  ┌─────────────────────────────────┐    │
│  │         CPU 限制                 │    │
│  │   - cpu.shares: CPU 权重        │    │
│  │   - cpu.cfs_period_us: 周期     │    │
│  │   - cpu.cfs_quota_us: 配额      │    │
│  └─────────────────────────────────┘    │
│  ┌─────────────────────────────────┐    │
│  │         内存限制                 │    │
│  │   - memory.limit_in_bytes       │    │
│  │   - memory.swap.limit_in_bytes  │    │
│  └─────────────────────────────────┘    │
│  ┌─────────────────────────────────┐    │
│  │         I/O 限制                 │    │
│  │   - blkio.weight: 权重          │    │
│  │   - blkio.throttle.read_bps    │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

### 3. Union File System（联合文件系统）

Docker 支持多种存储驱动：

| 存储驱动 | 适用文件系统 | 特点 |
|---------|-------------|------|
| `overlay2` | overlay2 | 推荐，性能最好 |
| `aufs` | aufs | 早期默认，已弃用 |
| `btrfs` | btrfs | 支持快照 |
| `devicemapper` | devicemapper | 块存储 |
| `vfs` | vfs | 兼容性好，性能差 |
| `zfs` | zfs | 支持快照和压缩 |

查看当前存储驱动：

```bash
docker info | grep "Storage Driver"
```

## Docker 的应用场景

### 1. 应用部署

```yaml
# 传统部署流程
开发环境编码 → 测试环境测试 → 生产环境部署
     ↓              ↓              ↓
  可能不同环境配置，导致"在我机器上能运行"问题

# Docker 部署流程
开发环境编码 → 构建镜像 → 测试环境运行镜像 → 生产环境运行镜像
     ↓                           ↓                    ↓
  环境完全一致，彻底解决环境差异问题
```

```
                      ▼
```

### 4. 开发环境标准化

```bash
# 无需在本地安装各种开发环境
# 一键启动完整的开发环境

# Python 开发环境

# Go 开发环境
```

## 最佳实践建议

### 1. 镜像构建原则

- 使用官方基础镜像
- 选择 Alpine 版本减小体积
- 多阶段构建减小最终镜像大小
- 不在镜像中存储敏感信息

### 2. 容器运行原则

- 一个容器只运行一个进程
- 使用 `.dockerignore` 排除不必要文件
- 使用 Docker Compose 管理多容器应用
- 设置合理的资源限制

### 3. 安全原则

- 以非 root 用户运行容器
- 限制容器能力（capabilities）
- 使用只读文件系统
- 定期扫描镜像漏洞
